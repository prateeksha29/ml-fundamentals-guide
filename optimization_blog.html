Why This Matters in Interviews

Optimization algorithms are the beating heart of machine learning. In interviews at companies like OpenAI, Anthropic, Google, Meta, and Netflix, you'll be expected to not just use these algorithms, but to deeply understand their mechanics, trade-offs, and implementation details.

Interviewers don't just want to hear "Adam is better than SGD." They want to know why, when, and how much better. They'll ask you to implement these from scratch, explain convergence properties, and diagnose why a model isn't training properly.

Interview Pattern Recognition

When asked "Why did you choose this optimizer?", never say "because it's standard" or "everyone uses it." Instead, demonstrate understanding: "I chose Adam because this problem has sparse gradients from the embedding layer, and Adam's adaptive learning rates handle that better than SGD. However, I'm monitoring for overfitting since Adam can sometimes converge to sharper minima."

The Core Algorithms

1. Gradient Descent (GD)

Foundation

The baseline. GD computes the gradient using the entire dataset before making a single update. It's deterministic, smooth, but impractical for large datasets.

θ = θ - η ∇J(θ) Where: θ = parameters η = learning rate ∇J(θ) = gradient of loss w.r.t. parameters

✓ Strengths

  • Guaranteed convergence to global minimum (convex)
  • Stable, smooth updates
  • Easy to analyze theoretically

✗ Weaknesses

  • Computationally expensive (full dataset each step)
  • Memory intensive
  • No online learning capability
  • Can't escape local minima in non-convex landscapes

Interview Question

"When would you actually use batch GD?" Answer: Small datasets where you can fit everything in memory, or when you need deterministic, reproducible results. Example: final fine-tuning of a model where stability matters more than speed.

2. Stochastic Gradient Descent (SGD)

Essential

The workhorse. SGD updates parameters after each training example (or small batch). This introduces noise, but that noise is actually helpful for escaping local minima.

For each sample (or mini-batch): θ = θ - η ∇J(θ; x⁽ⁱ⁾, y⁽ⁱ⁾) Mini-batch (typical): θ = θ - η (1/m) Σ ∇J(θ; x⁽ⁱ⁾, y⁽ⁱ⁾) where m = batch size (e.g., 32, 64, 256)

✓ Strengths

  • Fast updates, scales to massive datasets
  • Can escape local minima (noise helps)
  • Enables online learning
  • Lower memory footprint

✗ Weaknesses

  • Noisy convergence (oscillates near minimum)
  • Sensitive to learning rate choice
  • Same learning rate for all parameters
  • Can be slow on ill-conditioned problems
Critical Insight: The "stochastic" in SGD isn't a bug—it's a feature. That noise helps the optimizer explore the loss landscape and find flatter, more generalizable minima. This is why SGD often generalizes better than adaptive methods in vision tasks.

3. SGD with Momentum

Popular

Physics-inspired acceleration. Momentum accumulates a velocity vector that dampens oscillations and accelerates convergence in consistent gradient directions. Think of it as a ball rolling down a hill.

v = β·v + ∇J(θ) θ = θ - η·v Where: v = velocity (exponential moving average of gradients) β = momentum coefficient (typically 0.9)

✓ Strengths

  • Faster convergence than vanilla SGD
  • Dampens oscillations in high-curvature directions
  • Helps escape small local minima
  • Well-suited for deep networks

✗ Weaknesses

  • Introduces another hyperparameter (β)
  • Can overshoot minima if momentum too high
  • Still uses global learning rate

Common Follow-up

"What's Nesterov momentum?" It's a variant that computes the gradient at the "look-ahead" position: v = β·v + ∇J(θ - η·β·v). This anticipates where momentum will take you, leading to better-informed updates. Used in many production systems.

4. RMSProp

Adaptive

Adaptive learning rates per parameter. RMSProp (Root Mean Square Propagation) adapts the learning rate for each parameter based on the magnitude of recent gradients. Invented by Geoffrey Hinton, often not published formally but widely used.

E[g²] = β·E[g²] + (1-β)·g² θ = θ - (η / √(E[g²] + ε))·g Where: E[g²] = exponential moving average of squared gradients β = decay rate (typically 0.9) ε = small constant for numerical stability (1e-8) g = current gradient

✓ Strengths

  • Adapts learning rate per parameter
  • Works well with sparse gradients (NLP tasks)
  • Reduces need for manual learning rate tuning
  • Handles non-stationary objectives well

✗ Weaknesses

  • Can converge to suboptimal solutions
  • Learning rate still needs tuning
  • No bias correction in early iterations
Why RMSProp matters: It was one of the first optimizers to show that adaptive per-parameter learning rates could work in practice. It paved the way for Adam. You'll still see it in RNN and RL codebases.

5. Adam (Adaptive Moment Estimation)

Industry Standard

The default choice. Adam combines momentum (first moment) with RMSProp (second moment) and adds bias correction. It's robust, requires minimal tuning, and works across a wide range of problems.

m = β₁·m + (1-β₁)·g // First moment (momentum) v = β₂·v + (1-β₂)·g² // Second moment (RMSProp) m̂ = m / (1 - β₁ᵗ) // Bias correction v̂ = v / (1 - β₂ᵗ) θ = θ - η·m̂ / (√v̂ + ε) Default hyperparameters: β₁ = 0.9, β₂ = 0.999, ε = 1e-8, η = 1e-3

✓ Strengths

  • Works well out-of-the-box (robust defaults)
  • Combines benefits of momentum + adaptive LR
  • Bias correction handles early iterations properly
  • Efficient, low memory overhead
  • Excellent for sparse gradients and NLP

✗ Weaknesses

  • Can converge to sharper minima (worse generalization)
  • May not converge in some theoretical settings
  • Higher computational cost than SGD
  • Can overfit on small datasets

Red Flag Answer vs. Strong Answer

❌ Weak: "Adam is the best optimizer, everyone uses it."

✅ Strong: "I chose Adam for this transformer model because: (1) sparse gradients from masked attention benefit from adaptive LR, (2) β₂=0.999 provides stable second-moment estimates critical for large embedding matrices, (3) the warmup schedule prevents early divergence. However, I'm also testing AdamW to address potential weight decay coupling issues."

6. AdamW

Modern Standard

Adam with decoupled weight decay. The key insight: in Adam, L2 regularization and weight decay are NOT equivalent (unlike in SGD). AdamW fixes this by decoupling weight decay from the adaptive gradient computation.

m = β₁·m + (1-β₁)·g v = β₂·v + (1-β₂)·g² m̂ = m / (1 - β₁ᵗ) v̂ = v / (1 - β₂ᵗ) θ = θ - η·(m̂/(√v̂ + ε) + λ·θ) // Weight decay applied directly Where λ = weight decay coefficient (separate from LR)

✓ Strengths

  • Better generalization than Adam
  • Proper weight decay implementation
  • Decouples regularization from optimization
  • Standard in transformer training (BERT, GPT)

✗ Weaknesses

  • Requires tuning both η and λ
  • Slightly more complex than Adam
When it clicked: In Adam, L2 regularization gets scaled by the adaptive learning rate. Parameters with large gradients get less regularization. AdamW fixes this by applying weight decay directly to parameters, independent of gradients. This is why modern LLMs use AdamW, not Adam.

Quick Comparison Table

Algorithm Learning Rate Memory Best For When to Use
GD Global, fixed O(p) Small datasets Convex problems, deterministic results needed
SGD Global, fixed O(p) Vision (CNNs) When generalization > speed, large datasets
SGD+Momentum Global, fixed O(2p) Deep networks ResNets, when oscillations are a problem
RMSProp Per-parameter, adaptive O(2p) RNNs, RL Non-stationary problems, sparse gradients
Adam Per-parameter, adaptive O(3p) Transformers, NLP Default choice, sparse data, need fast convergence
AdamW Per-parameter, adaptive O(3p) LLMs, transformers When proper regularization matters (BERT, GPT)

* p = number of parameters in the model

Implementation Questions You'll Face

Question 1: "Implement Adam from scratch in NumPy"

import numpy as np class Adam: def __init__(self, lr=1e-3, beta1=0.9, beta2=0.999, eps=1e-8): self.lr = lr self.beta1 = beta1 self.beta2 = beta2 self.eps = eps self.m = None # First moment self.v = None # Second moment self.t = 0 # Time step def update(self, params, grads): # Initialize moments on first call if self.m is None: self.m = np.zeros_like(params) self.v = np.zeros_like(params) self.t += 1 # Update biased moments self.m = self.beta1 * self.m + (1 - self.beta1) * grads self.v = self.beta2 * self.v + (1 - self.beta2) * (grads ** 2) # Bias correction m_hat = self.m / (1 - self.beta1 ** self.t) v_hat = self.v / (1 - self.beta2 ** self.t) # Update parameters params -= self.lr * m_hat / (np.sqrt(v_hat) + self.eps) return params

Follow-up Questions

Why bias correction? In early iterations, m and v are biased toward zero (initialized at zero). Without correction, updates would be too small initially. Bias correction term (1 - βᵗ) approaches 0 as t increases, removing the correction after warmup.

What if we skip bias correction? You'd need a warmup phase (gradually increasing LR) to compensate. Some implementations do this instead.

Question 2: "Why does SGD sometimes outperform Adam on vision tasks?"

The answer lies in the geometry of minima: Adam's adaptive learning rates can lead to sharper minima (high curvature). These generalize worse because small perturbations to inputs cause large changes in loss. SGD, especially with momentum, tends to find flatter minima with better generalization. This is why ResNets trained with SGD often beat Adam-trained ones, despite Adam converging faster.

Question 3: "Your transformer isn't converging. How do you debug?"

Systematic debugging approach:

  • Check gradient norms: Are they exploding? Vanishing? Use gradient clipping.
  • Learning rate: Start with 1e-4, use warmup (linear increase for first ~5% of steps), then cosine decay.
  • Batch size: Larger batches need higher LR (linear scaling rule: LR ∝ batch_size).
  • Adam parameters: For transformers, β₂=0.98 sometimes works better than 0.999.
  • Weight initialization: Use Xavier/He initialization. Bad init can kill Adam.
  • Check for NaNs: Mixed precision training can cause instability. Use loss scaling.

The Learning Rate Question

No optimizer discussion is complete without learning rate scheduling. Here's what interviewers expect you to know:

Learning Rate Schedules

1. Step Decay: Reduce LR by factor (e.g., 0.1) every N epochs. Simple, interpretable.

η = η₀ × γ^(epoch / step_size)

2. Cosine Annealing: Smooth reduction following cosine curve. Popular in transformers.

η = η_min + 0.5 × (η_max - η_min) × (1 + cos(πt / T))

3. Linear Warmup + Cosine: The modern standard for LLMs.

Warmup (0 to warmup_steps): η = η_max × (t / warmup_steps) Decay (warmup_steps to T): Use cosine annealing

4. One Cycle: Fast.ai's contribution. Increase then decrease LR.

Interview Gold

"Why warmup?" In early training, parameter estimates are random. Large learning rates can cause divergence. Warmup gives the optimizer time to find a reasonable region of parameter space before full-speed optimization. Critical for Adam with large models (GPT, BERT).

Real Interview Scenarios

Scenario 1: Anthropic System Design Interview

Question: "You're training a 70B parameter LLM. Walk me through your optimization strategy."

Strong answer structure:
  • Optimizer choice: AdamW with β₁=0.9, β₂=0.95, ε=1e-8. Lower β₂ than default because of large model size.
  • Learning rate: Peak LR ~1e-4 (found via short LR range test). Linear warmup over 2000 steps, then cosine decay to 1e-5.
  • Batch size strategy: Start with effective batch size of 4M tokens (gradient accumulation). Monitor gradient variance.
  • Gradient clipping: Clip by global norm at 1.0 to prevent instability.
  • Mixed precision: bfloat16 for forward/backward, fp32 for master weights and optimizer states.
  • Monitoring: Track loss, gradient norms, learning rate, parameter norms per layer. Watch for layer collapse.

Scenario 2: Google Coding Round

Question: "Implement SGD with momentum. Then explain how you'd parallelize it across multiple GPUs."

Two-part answer

Part 1: Implement the algorithm (similar to Adam example above).

Part 2 (parallelization): Data parallelism is standard. Each GPU computes gradients on its data shard, then all-reduce to average gradients before the optimizer step. Momentum state stays local (no communication needed). Critical: gradient averaging happens BEFORE momentum update. Key detail: with N GPUs and batch size B, effective batch size is N×B, so scale LR by √N (not N linearly).

Final Interview Prep Checklist

✓ Must-Know Concepts

  • Implement SGD, SGD+Momentum, Adam from scratch in NumPy
  • Explain why Adam needs bias correction (and prove it mathematically)
  • Know when SGD outperforms Adam (vision) and vice versa (NLP)
  • Understand the difference between Adam and AdamW
  • Explain learning rate warmup and why it matters
  • Debug convergence issues systematically
  • Know the memory footprint of each optimizer
  • Understand gradient clipping (by value vs. by norm)
  • Explain the relationship between batch size and learning rate
  • Be ready to discuss second-order methods (L-BFGS, natural gradient)

The Meta-Question

At the end, interviewers often ask: "If you could design a new optimizer, what would you improve?" This tests creative thinking. Good answer: "I'd focus on adaptive batch sizing—dynamically adjusting batch size based on gradient variance. High variance → larger batches for stable estimates. Low variance → smaller batches for faster iterations. This could auto-tune the exploration-exploitation trade-off."
