1. Flash Attention: Making Attention Fast Without Approximation

📄
FlashAttention: Fast and Memory-Efficient Exact Attention with IO-Awareness
Dao, Fu, Ermon, Rudra & Ré · NeurIPS 2022

Standard attention materializes the full \(n \times n\) attention matrix in GPU high-bandwidth memory (HBM). Flash Attention's insight: attention is memory-bound, not compute-bound. Modern GPUs have enormous compute throughput (312 TFLOPS on A100) but relatively slow memory bandwidth. The bottleneck isn't the arithmetic — it's reading and writing the large intermediate matrices to and from memory.

1.1 The GPU Memory Hierarchy

Figure 1: GPU memory hierarchy. SRAM is ~10× faster than HBM but ~2000× smaller. Standard attention reads/writes intermediate matrices to HBM repeatedly. Flash Attention keeps computation in SRAM.

1.2 The Problem with Standard Attention

Standard attention computes three intermediate matrices that must be stored in HBM:

Figure 2: Standard attention writes two n×n intermediate matrices (S and P) to slow HBM. Flash Attention computes attention in tiles within fast SRAM, never materializing the full matrices.

1.3 How Flash Attention Works: Tiling + Online Softmax

Flash Attention splits Q, K, V into small blocks that fit in SRAM. It loads one block of K, V into SRAM, then iterates over blocks of Q, computing the attention output tile-by-tile. The challenge: softmax requires the full row to normalize. Flash Attention uses the online softmax trick — it keeps running statistics (max and sum) that are updated as each block is processed, giving the exact same result as materializing the full matrix.

Figure 3: Flash Attention tiling. Blocks of Q, K, V are loaded into fast SRAM. Attention is computed tile-by-tile with running softmax statistics, then the output block is written back to HBM. The full n×n matrix never exists.
⚡ The key results
  • Exact — produces identical output to standard attention. No approximation, no quality loss.
  • Memory: \(O(n)\) instead of \(O(n^2)\) — the n×n matrix is never stored.
  • Speed: 2-4× faster wall-clock despite same FLOPs (even slightly more due to recomputation in backward pass) — because fewer slow HBM reads/writes.
  • HBM accesses: \(O(N^2 d^2 M^{-1})\) vs standard \(O(Nd + N^2)\), where M is SRAM size.
  • Longer sequences: enables 4-16× longer context on the same GPU.
💡 Why this matters conceptually

Flash Attention proved that the attention bottleneck wasn't computation but memory bandwidth. Approximate attention methods (Linformer, Performer) tried to reduce FLOPs by changing the math — but often didn't achieve wall-clock speedup because they didn't address the IO bottleneck. Flash Attention keeps the exact same math but reorganizes where the computation happens in the memory hierarchy. This is a systems insight, not an algorithmic one — and it's now the default attention implementation in virtually all production models.

2. KV-Cache: Avoiding Redundant Computation at Inference

2.1 The Problem

During autoregressive generation, at step \(t\) the decoder must attend to all positions 1 through \(t\). Without caching, this means recomputing K and V for every previous token at every step — pure waste, since token 1's K and V never change.

Figure 4: Without caching, previous K,V are recomputed at every step (O(n²) total work). With KV-cache, each step only processes one new token and appends to the cache (O(n) total).

2.2 Memory Cost

KV-Cache Memory per Token
$$\text{KV per token} = 2 \times n_{\text{layers}} \times n_{\text{kv\_heads}} \times d_{\text{head}} \times \text{bytes}$$

For Llama 2 70B (80 layers, 8 KV heads with GQA, d_head=128, float16):

  • Per token: \(2 \times 80 \times 8 \times 128 \times 2 = 320\) KB
  • 4096-token context: 1.3 GB per sequence
  • Batch of 32: ~42 GB just for KV-cache

This is often the inference bottleneck — more GPU memory goes to KV-cache than model weights. This is why the following optimizations exist:

2.3 PagedAttention (vLLM)

Standard KV-cache pre-allocates a contiguous block of memory for the maximum sequence length per request. If a request uses only 200 tokens of a 4096-token allocation, the remaining 3896 tokens worth of memory is wasted. Across hundreds of concurrent requests, this fragmentation wastes 60-80% of GPU memory.

PagedAttention (Kwon et al., 2023, from the vLLM project at UC Berkeley) borrows the idea of virtual memory paging from operating systems. Instead of contiguous allocation, the KV-cache is divided into fixed-size blocks (pages). Each sequence's KV-cache is stored as a chain of non-contiguous pages, allocated on demand as the sequence grows. A page table maps logical positions to physical memory blocks.

💡 Why this matters for production

PagedAttention eliminates fragmentation, achieving near-zero waste. This enables 2-4× higher throughput (more concurrent requests per GPU) with the same hardware. It also enables prefix caching — if multiple requests share the same system prompt, they can share KV-cache pages for that prefix (copy-on-write, like OS fork). At Amazon scale serving millions of users, this is the difference between needing 100 GPUs vs 30 GPUs. vLLM with PagedAttention is now the dominant serving framework for LLMs.

2.4 KV-Cache Quantization

Store cached K and V vectors in int8 or int4 instead of float16. This halves or quarters the cache memory with minimal quality impact — K and V values tend to be well-distributed and quantize cleanly. Combined with GQA, you can achieve 8-16× cache reduction (4× from GQA × 2-4× from quantization), making long-context serving feasible on limited hardware.

3. Multi-Query & Grouped-Query Attention: Shrinking the KV-Cache

Full multi-head attention (MHA) gives each head its own K and V projections. The KV-cache stores separate K,V for every head — expensive.

Figure 5: MHA vs MQA vs GQA. MHA: each head has unique K,V (most memory). MQA: all heads share one K,V (8× cache reduction, some quality loss). GQA: groups of heads share K,V (middle ground — used in Llama 2/3, Mistral).

4. Mixture of Experts: Scaling Parameters Without Scaling Compute

📄
Outrageously Large Neural Networks: The Sparsely-Gated Mixture-of-Experts Layer
Shazeer et al. · ICLR 2017 (applied to Transformers in Switch Transformer, Fedus et al. 2022)

In a standard Transformer, every token passes through the same FFN — all parameters are used for every token. MoE replaces the single FFN with multiple expert FFNs plus a gating/router network that selects which experts to activate for each token. Only top-k experts (typically k=1 or k=2) process each token.

Figure 6: Standard FFN (all tokens through same FFN, 100% params active) vs MoE (router selects top-k experts per token, ~25% params active). MoE scales capacity without proportional compute increase.

4.1 How the Router Works

The router is surprisingly simple — a single linear layer that maps each token's representation to a probability distribution over experts:

Router / Gating Network
$$g(x) = \text{softmax}(W_g \cdot x) \quad W_g \in \mathbb{R}^{d_{\text{model}} \times N_{\text{experts}}}$$ $$\text{output} = \sum_{i \in \text{top-}k} g_i(x) \cdot E_i(x)$$

For each token, the router produces a score for every expert, selects the top-k (typically k=1 or k=2), and the token is processed by only those experts. The final output is a weighted sum of the selected experts' outputs, weighted by their router probabilities. Tokens that weren't selected for an expert don't pass through it at all — the expert's parameters are not activated.

Token dropping: Each expert has a capacity limit (e.g., tokens per expert = batch_size × seq_len / num_experts × capacity_factor). When an expert is full, excess tokens are dropped — they bypass the FFN and pass through via the residual connection only. This prevents load imbalance from becoming a latency bottleneck (the slowest expert determines throughput). The capacity_factor (typically 1.0-1.5) controls the tradeoff between dropping tokens and wasting compute on empty expert slots.

4.2 The Gating Collapse Problem

The router can learn to send all tokens to the same expert — gating collapse. One expert gets all the traffic, others are wasted. This is solved with a load-balancing auxiliary loss that penalizes uneven expert utilization:

Load-Balancing Loss (Switch Transformer)
$$\mathcal{L}_{\text{balance}} = \alpha \cdot N \sum_{i=1}^{N} f_i \cdot P_i$$

where \(f_i\) = fraction of tokens routed to expert \(i\), \(P_i\) = average router probability for expert \(i\), \(\alpha\) = balancing coefficient (typically 0.01). This is minimized when all experts receive equal traffic.

⚠️ Connection to your work

In your MFAR project, you encountered field weight collapse — your query-side gating layer would assign all weight to one metadata field, ignoring others. You fixed this with an MoE-inspired load-balancing loss. This is the exact same problem and solution — be ready to draw this parallel explicitly in the interview. The interviewer will likely ask: "Have you encountered this gating problem before?"

5. Modern Positional Encoding: RoPE and ALiBi

5.1 RoPE — Rotary Position Embedding (Su et al., 2021)

📄
RoFormer: Enhanced Transformer with Rotary Position Embedding
Su, Lu, Pan, Murtadha, Wen & Liu · arXiv 2021 · Used in Llama, GPT-NeoX, PaLM

Instead of adding positional information to embeddings (sinusoidal), RoPE rotates the Q and K vectors by position-dependent angles. The key property: the dot product between rotated Q and K depends only on relative position \((m - n)\), not absolute positions \(m\) and \(n\).

Figure 7: RoPE rotates Q and K vectors by position-dependent angles. The dot product between two rotated vectors depends only on the angle difference (relative position m-n), not the absolute positions. Each dimension pair rotates at a different frequency, similar to sinusoidal PE.
RoPE Rotation (for one dimension pair)
$$\text{RoPE}(x_m^{(1)}, x_m^{(2)}, m) = \begin{pmatrix} \cos m\theta & -\sin m\theta \\ \sin m\theta & \cos m\theta \end{pmatrix} \begin{pmatrix} x_m^{(1)} \\ x_m^{(2)} \end{pmatrix}$$ $$\theta_i = 10000^{-2i/d} \quad \text{(same base as sinusoidal PE)}$$

Key properties of RoPE:

  • Relative position encoded through rotation — \(\langle \text{RoPE}(q, m),\, \text{RoPE}(k, n) \rangle\) is a function of \(m - n\) only
  • Norm-preserving — rotation doesn't change vector magnitude, preserving semantic information
  • No extra parameters — deterministic like sinusoidal, but applied to Q and K (not added to input)
  • Decaying dependency — attention naturally decreases with distance (desirable for language)
  • Applied at every layer — unlike sinusoidal which is added once at input; overhead is negligible

Context length extension with RoPE: Since the rotation angles are \(m\theta_i\), positions beyond training length produce unseen angles — the model encounters rotations it never saw during training, causing quality degradation.

Position Interpolation (Chen et al., 2023): The simplest fix — linearly scale down position indices so that the extended range maps back to the training range. For a model trained at 4K extending to 32K: position \(m\) becomes \(m \times 4096/32768\). All angles stay within the trained range. Requires a small amount of fine-tuning to adapt.

NTK-aware interpolation (Reddit/bloc97, 2023): Instead of scaling all frequencies equally (which destroys high-frequency position information), NTK-aware scaling increases the base frequency: \(\theta_{\text{base}} = 10000 \rightarrow 10000 \times \alpha\) where \(\alpha\) is the extension ratio. This compresses low-frequency (long-range) rotations more than high-frequency (short-range) ones, preserving local position resolution. Can work without fine-tuning.

YaRN (Peng et al., 2023): Combines NTK-aware scaling with a learned temperature factor that adjusts attention logit scale for extended contexts. Currently the most effective method — Llama models use YaRN-style extension to go from 4K training to 128K+ context.

5.2 ALiBi — Attention with Linear Biases (Press et al., 2022)

📄
Train Short, Test Long: Attention with Linear Biases Enables Input Length Generalization
Press, Smith & Lewis · ICLR 2022 · Used in BLOOM, MPT

ALiBi takes the simplest possible approach: no positional encoding at all. Instead, add a linear penalty to attention scores based on distance:

ALiBi
$$\text{attention}_{ij} = \text{softmax}\!\left(\frac{q_i \cdot k_j}{\sqrt{d_k}} - m \cdot |i - j|\right)$$

where \(m\) is a head-specific slope (fixed, not learned). Closer tokens get higher scores, distant tokens are penalized linearly.

Advantages: Extrapolates naturally to any length (linear bias extends trivially), zero additional parameters, simple to implement. Tradeoff: Less flexible than RoPE for tasks requiring precise long-range retrieval.

5.3 Sliding Window Attention (Mistral)

Each token only attends to the last \(w\) tokens (e.g., \(w = 4096\)) instead of all previous positions. Complexity drops from \(O(n^2)\) to \(O(n \cdot w)\). Information beyond the window still propagates through stacked layers — layer 2's window "sees" tokens that layer 1's window processed, effectively doubling the receptive field per layer. With \(L\) layers and window \(w\), the effective receptive field is \(L \times w\).

6. Speculative Decoding: Parallelizing Inference

Autoregressive decoding is inherently sequential — one token at a time. Speculative decoding exploits the fact that verification is cheaper than generation:

Figure 8: Speculative decoding. A small draft model proposes k tokens cheaply. The large model verifies all k in one parallel forward pass. Correct prefix is accepted; the first wrong token is resampled from the big model. Exact same output distribution as the big model alone.

6.1 Medusa: Parallel Decoding Without a Draft Model

📄
Medusa: Simple LLM Inference Acceleration Framework with Multiple Decoding Heads
Cai et al. · 2024

Instead of using a separate draft model, Medusa adds multiple lightweight prediction heads to the base model itself. Head 1 predicts token \(t+1\) (same as normal), head 2 predicts \(t+2\), head 3 predicts \(t+3\) — all in a single forward pass. Each extra head is just a few linear layers trained on top of the frozen base model.

The candidates from all heads form a tree of possibilities. A tree-attention verification step checks which paths are correct. Compared to speculative decoding: no separate draft model to load into memory, simpler deployment, but slightly less flexible (the extra heads must be trained for each model).

7. State Space Models: The Alternative Architecture

📄
Mamba: Linear-Time Sequence Modeling with Selective State Spaces
Gu & Dao · 2023 · Parallel training + O(1) inference per step

Instead of fixing Transformer inference, Mamba replaces attention entirely with selective state space layers. The key innovation: the same computation can be expressed as either a parallel scan (for training, like Transformers) or a recurrence (for inference, like RNNs).

TransformerRNNMamba / SSM
TrainingO(1) parallel ✓O(n) sequential ✗O(1) parallel ✓
Inference per stepO(t·d) attention + KV cacheO(d²) fixedO(d) fixed ✓
Inference memoryKV-cache grows with nFixed hidden stateFixed state ✓
Long-range retrievalDirect O(1) lookup ✓Degraded (compressed)Good but not exact

Mamba gets the best of both worlds for training speed and inference cost. The tradeoff: attention's ability to do arbitrary lookup over the full context (token 500 directly accessing token 3 with full fidelity) is hard to match with a fixed-size state. Mamba works well for many tasks but can lag on tasks requiring precise long-range retrieval. Hybrid architectures (some attention layers + some Mamba layers) are an active area.

10. Interview Questions

Q1 — Flash Attention
What is Flash Attention and why is it faster despite having the same (or more) FLOPs?
Flash Attention is an IO-aware exact attention algorithm. It uses tiling to compute attention in blocks within fast on-chip SRAM, avoiding materializing the full n×n attention matrix in slow HBM. Standard attention is memory-bound, not compute-bound — the bottleneck is reading/writing intermediate matrices to HBM, not the arithmetic. Flash Attention reduces HBM accesses from \(O(Nd + N^2)\) to \(O(N^2 d^2 M^{-1})\), achieving 2-4× wall-clock speedup and reducing memory from O(n²) to O(n). It produces exact results — no approximation.
Q2 — Flash Attention
How does Flash Attention handle softmax if it never materializes the full attention matrix?
The online softmax trick. Standard softmax needs the full row to compute the normalization denominator. Flash Attention maintains running statistics — a running maximum \(m\) and running sum \(\ell\) — that are updated as each block of K is processed. When rescaling between blocks, it adjusts the partial outputs by the ratio of old and new normalization factors. After processing all blocks, the statistics contain the exact softmax denominator, giving identical results to standard attention.
Q3 — KV-Cache
What is KV-cache and why is it necessary for efficient Transformer inference?
During autoregressive generation, the K and V vectors for previous tokens never change — only the new token's Q, K, V need computation. KV-cache stores the K and V from all previous positions, so at step \(t\) you only run 1 token through the model (not \(t\) tokens). This reduces total work from O(n²) to O(n) across all generation steps. The tradeoff: cache grows linearly with sequence length — for Llama 70B at 4096 tokens, that's ~1.3 GB per sequence, often exceeding the model weights as the memory bottleneck in production serving.
Q4 — GQA
What's the difference between MHA, MQA, and GQA?
MHA: every head has its own K, V projections — maximum expressiveness, maximum KV-cache cost. MQA (Shazeer 2019): all heads share a single K, V — cache shrinks by factor of h (e.g., 8×), but quality drops noticeably. GQA (Ainslie et al. 2023): groups of heads share K, V (e.g., 8 Q heads, 2 KV groups) — 4× cache reduction with minimal quality loss. GQA is the dominant choice in modern LLMs (Llama 2/3, Mistral) because it hits the best tradeoff between memory and quality.
Q5 — MoE
How does Mixture of Experts work in Transformers? What problem does it solve?
MoE replaces the single FFN with multiple expert FFNs + a learned router/gating network. For each token, the router selects top-k experts (typically k=2). Only those experts process the token; the rest are inactive. This decouples total model capacity (all expert parameters) from per-token compute (only k expert parameters). Mixtral 8×7B has 47B total params but only activates 13B per token — running like a 13B model while having the knowledge capacity of a much larger one.
Q6 — MoE + Your Work
What is gating collapse in MoE and how is it fixed?
Gating collapse is when the router learns to send all tokens to the same expert, leaving others unused. It occurs because the rich-get-richer dynamic — a frequently-used expert gets better gradients, making the router prefer it more. Fixed with a load-balancing auxiliary loss: \(\mathcal{L}_{\text{balance}} = \alpha \cdot N \sum f_i \cdot P_i\), which penalizes uneven expert utilization. I encountered the same problem in my MFAR work — query-side field gating would collapse to one metadata field. I used an MoE-inspired load-balancing loss to distribute weights across fields, which was critical for achieving the +30% recall improvement on ADO connectors.
Q7 — RoPE
How does RoPE encode positional information differently from sinusoidal PE?
Sinusoidal PE adds a fixed vector to the input embedding at the first layer — position and semantics are mixed. RoPE rotates Q and K vectors by position-dependent angles at every layer. The key insight: the dot product between two rotated vectors depends only on the angle difference (relative position \(m - n\)), not absolute positions. This naturally gives relative position encoding. RoPE preserves vector norms (semantics untouched), requires no extra parameters, and enables context length extension through angle rescaling (NTK interpolation, YaRN).
Q8 — ALiBi
What is ALiBi and when would you choose it over RoPE?
ALiBi adds a linear distance penalty directly to attention scores: \(\text{score}_{ij} = q_i \cdot k_j / \sqrt{d_k} - m \cdot |i-j|\). No positional encoding at all — position is purely in the attention bias. It's simpler to implement, has zero extra parameters, and extrapolates to any length trivially (the linear bias extends naturally). Choose ALiBi over RoPE when length extrapolation without fine-tuning is critical and you don't need precise long-range retrieval. RoPE is more flexible and dominant in practice (Llama, PaLM), while ALiBi is used in BLOOM and MPT.
Q9 — Speculative Decoding
How does speculative decoding speed up inference without losing quality?
A small draft model generates k candidate tokens quickly. The large model then verifies all k tokens in one parallel forward pass (possible because verification is just checking — compute the logits for all k positions simultaneously). Accept the longest correct prefix; resample the first wrong token from the large model's distribution. The output distribution is mathematically identical to the large model alone — zero quality loss. Typical speedup: 2-3×. The key insight: verification (parallel) is cheaper than generation (sequential).
Q10 — Mamba/SSM
What are State Space Models and how do they compare to Transformers?
SSMs (like Mamba) replace attention with selective state space layers that can be computed as either a parallel scan (training — like Transformers) or a recurrence (inference — like RNNs). This gives parallel training with O(1)-per-step inference and fixed memory — no KV-cache growth. The tradeoff: attention can do arbitrary lookup over the full context (any token directly accesses any other), which is hard to replicate with a fixed-size state. Mamba works well for many tasks but can lag on tasks requiring precise long-range retrieval. Hybrid architectures mixing attention and SSM layers are an active research area.
Q11 — Systems
If you were deploying a recommendation model at Prime Video scale, which of these techniques would matter most?
For a retrieval model (encoding user histories): Flash Attention for training efficiency on long watch histories. Sliding window attention if histories are very long (thousands of items) — recent items matter most. RoPE for temporal position encoding.

For serving at scale: KV-cache with GQA to serve millions of concurrent users within GPU memory budgets. Speculative decoding if using an autoregressive model for generation (e.g., generating explanations for recommendations). MoE if different content types (movies vs TV vs sports) benefit from specialized expert FFNs.

The overarching principle: at Amazon scale, latency and memory are the constraints, not model quality in isolation. Every technique here is about getting Transformer-quality results within production budgets.
Q12 — Comparison
Compare Flash Attention, sparse attention, and linear attention. When would you use each?
Flash Attention: exact attention, same O(n²) FLOPs but massively fewer HBM accesses → 2-4× wall-clock speedup. Use it always as the default — no quality tradeoff. Sparse attention (BigBird, Longformer): predefined patterns (local window + global tokens), O(n·k) complexity. Use when sequences are very long (documents, genomics) and local structure dominates. Quality loss on tasks needing full global attention. Linear attention (Performer, Katharopoulos): approximate softmax with kernels, O(n) complexity. Theoretically attractive but in practice often underperforms full attention by a noticeable margin. Use only if O(n²) is truly infeasible even with Flash Attention. In practice: Flash Attention first, sparse if sequences are extremely long, linear rarely.
