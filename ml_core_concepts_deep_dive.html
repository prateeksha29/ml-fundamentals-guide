1. Linear Regression

Linear regression models the relationship between features X and a continuous target y as a linear function: ŷ = Xw + b. The goal is to find weights w and bias b that minimize the distance between predictions and actuals.

The Loss Function: MSE

We minimize the Mean Squared Error:

$$\mathcal{L} = \frac{1}{N}\sum_{i=1}^{N}(y_i - \hat{y}_i)^2$$

Why squared error? Squaring penalizes large errors more heavily, is differentiable everywhere (unlike absolute error), and yields a convex optimization landscape with a single global minimum for linear models.

MSE vs MAE: When to Use Which

MAE (Mean Absolute Error) penalizes all errors equally, making it robust to outliers — a single extreme prediction doesn't dominate the loss. MSE amplifies outlier influence quadratically. Use MAE when outliers are noise you want to ignore; use MSE when large errors are genuinely costly (e.g., financial forecasting where a $1M miss matters much more than a $100 miss). A practical note: MAE's gradient is discontinuous at zero, which can cause optimization issues — Huber loss combines the best of both (MSE near zero, MAE far from zero).

Closed-Form Solution (Normal Equation)

$$w^* = (X^TX)^{-1}X^Ty$$

This works by setting the gradient of MSE to zero and solving directly. But it requires inverting an (n_features × n_features) matrix, which is O(d³). For large feature dimensions or near-singular matrices, gradient descent or regularized approaches are preferred.

Feature Scaling & the Normal Equation
The normal equation doesn't require feature scaling — it finds the exact solution regardless of feature magnitudes. But gradient descent does — features on different scales create elongated loss contours, causing oscillation and slow convergence. Always standardize features before gradient-based optimization. More on this in Section 8.

Assumptions of Linear Regression

In a statistical context, OLS linear regression assumes: linearity (the true relationship is linear), independence of errors, homoscedasticity (constant error variance), no perfect multicollinearity among features, and normally distributed residuals (for inference, not prediction). In ML interviews, the key ones to mention are linearity, independence, and multicollinearity.

Common Gotcha
"Does linear regression assume features are normally distributed?" — No. The normality assumption applies to the residuals, not the features. And even that is only required for valid confidence intervals and p-values, not for the point estimates themselves.

R² and Adjusted R²

R² measures the proportion of variance explained: R² = 1 − (SS_res / SS_tot). It always increases (or stays flat) when you add features, even useless ones. Adjusted R² penalizes for additional features: it can decrease if a new feature doesn't improve the model enough to justify the added complexity. In interviews, always mention adjusted R² when discussing feature selection.

Interview Question
When would you use linear regression over a more complex model?

When interpretability matters (e.g., understanding which features drive predictions), when you have limited data relative to features (simpler models generalize better), when the relationship is approximately linear, or as a baseline before trying complex models. Linear regression also trains in seconds and serves in microseconds, which matters at scale.

Interview Question
What happens to linear regression when features are highly correlated (multicollinearity)?

The weight estimates become unstable — small changes in data cause large swings in coefficients. The matrix XTX becomes near-singular, so its inverse amplifies noise. The model still predicts fine (predictions are stable), but individual weights become uninterpretable. The fix: drop correlated features, use PCA, or apply regularization (Ridge regression specifically addresses this).

Interview Question
Can linear regression be used for non-linear relationships?

Yes — through feature engineering. You can add polynomial features (x², x³, x₁·x₂), log/sqrt transforms, or any non-linear transformation of the inputs. The model is still "linear" because it's linear in the parameters — ŷ = w₁x + w₂x² is linear regression on features [x, x²]. This is called polynomial regression. The risk is overfitting with too many engineered features.

Interview Question
When would you use MAE instead of MSE as your loss function?

Use MAE when your data has outliers that you want to be robust to — MSE squares errors, so a single outlier can dominate the loss and distort your model. MAE treats all errors linearly. The tradeoff: MAE produces a model that predicts the median of the target, while MSE predicts the mean. If you want the best of both, use Huber loss — it behaves like MSE for small errors (smooth, easy to optimize) and like MAE for large errors (robust to outliers). The transition point δ is a hyperparameter you tune.

Interview Question
Can R² be negative?

Yes — and this surprises most people who assume R² lives in [0, 1]. R² = 1 − (SS_res / SS_tot). If SS_res > SS_tot, meaning your model's predictions are worse than simply predicting the mean of y for every example, then R² goes negative. This can happen when you evaluate a model on a test set that it was not fit on, especially if the model is badly overfit or the test distribution has shifted. On the training set of an OLS model, R² is guaranteed ≥ 0 (you can't do worse than the mean when you're optimizing against that same data). But on unseen data, anything goes.

Interview Question
How do you interpret a coefficient of 0.3 on feature X in a linear regression?

"A one-unit increase in X is associated with a 0.3 increase in y, holding all other features constant." That last phrase is critical — and it's where things get tricky. If X is correlated with other features, the "holding constant" assumption is artificial: in reality, changing X also shifts the correlated features. The coefficient reflects the partial effect, not the total effect. With strong multicollinearity, individual coefficients can flip signs or inflate wildly, even though the model predicts well overall. If interpretability is the goal, consider standardizing features first so coefficients are on the same scale, and use techniques like SHAP values or partial dependence plots for more robust attribution.

2. Logistic Regression

Despite the name, logistic regression is a classification algorithm. It models the probability that an input belongs to class 1 by passing a linear combination through the sigmoid function:

$$P(y=1|x) = \sigma(w^Tx + b) = \frac{1}{1 + e^{-(w^Tx + b)}}$$

The sigmoid squashes any real number into (0, 1), giving us a valid probability. The decision boundary is the hyperplane where wTx + b = 0, which corresponds to P = 0.5 at default threshold.

The Loss: Binary Cross-Entropy

$$\mathcal{L} = -\frac{1}{N}\sum_{i=1}^{N}\Big[y_i\log(\hat{p}_i) + (1-y_i)\log(1-\hat{p}_i)\Big]$$

Why not MSE? MSE with sigmoid creates a non-convex loss landscape with many local minima. Cross-entropy is convex for logistic regression and has a probabilistic justification: minimizing BCE is equivalent to maximum likelihood estimation of the Bernoulli distribution parameters.

The Gradient: Elegant Simplicity

The gradient of BCE with respect to the logit z = wTx + b simplifies beautifully: ∂L/∂z = ŷ − y. The sigmoid's derivative and the log in cross-entropy cancel out perfectly, leaving just the difference between prediction and label. This means the gradient is large when the model is confidently wrong and small when it's close — exactly the behavior you want. The weight gradient follows: ∂L/∂w = (ŷ − y)·x.

Maximum Likelihood Interpretation

Cross-entropy isn't an arbitrary choice — it falls directly out of maximum likelihood estimation. If you model each label as a Bernoulli random variable with parameter p = σ(wTx), the log-likelihood of the data is exactly the negative of BCE. So minimizing cross-entropy = maximizing the likelihood that the model assigned to the observed labels. This is why logistic regression produces well-calibrated probabilities: it's directly optimizing the likelihood of the data under a probabilistic model.

Why the Sigmoid?

The sigmoid arises naturally from modeling log-odds as a linear function. If you assume log(P/(1−P)) = wTx + b (the log-odds are linear in features), then solving for P gives you exactly the sigmoid. It's not an arbitrary choice — it's the canonical link function for Bernoulli data in the generalized linear model framework.

Multiclass Extension: Softmax Regression

For K classes, replace sigmoid with softmax: P(y=k) = exp(z_k) / Σ_j exp(z_j), and use categorical cross-entropy as the loss. Each class gets its own weight vector. This is sometimes called multinomial logistic regression.

No Closed-Form Solution

Unlike linear regression, logistic regression has no closed-form solution — the cross-entropy loss with sigmoid doesn't yield a system of equations you can solve directly. You must use iterative optimization: gradient descent, Newton's method (used by statsmodels), or L-BFGS (sklearn's default). Newton's method converges in fewer iterations because it uses second-order curvature information, but each iteration is more expensive. For large datasets, SGD or L-BFGS is preferred.

Threshold ≠ 0.5
A common misconception: the optimal classification threshold is not always 0.5. For imbalanced classes (e.g., fraud detection with 0.1% positives), you tune the threshold using the precision-recall curve or by optimizing the F1 score. In an interview, always ask "what metric are we optimizing?" before picking a threshold.
Interview Question
Why is logistic regression considered a linear model if it uses a non-linear sigmoid function?

The decision boundary is linear — it's the hyperplane wTx + b = 0. The sigmoid just maps the linear output to a probability. The model partitions feature space with a flat hyperplane; it can't learn curved decision boundaries without feature engineering. That's what makes it "linear" — not the output transformation, but the geometry of the decision surface.

Interview Question
How do you handle class imbalance with logistic regression?

Several approaches, in rough priority order: (1) Adjust class weights in the loss function — weight the minority class higher so the model pays more attention to it. Most libraries support class_weight='balanced'. (2) Tune the decision threshold using PR curves instead of using 0.5. (3) Resample the data — oversample minority (SMOTE) or undersample majority. (4) Use evaluation metrics that handle imbalance: PR-AUC, F1, or MCC rather than accuracy. Approach (1) + (2) is usually the best starting point.

Interview Question
Can logistic regression output calibrated probabilities?

Logistic regression is naturally well-calibrated when its assumptions hold (the log-odds are linear in features) because it's optimizing the log-likelihood directly. This is one of its advantages over models like SVMs or random forests, which need post-hoc calibration (Platt scaling, isotonic regression). However, calibration degrades with strong regularization, feature mismatch, or distribution shift — always check with a calibration plot (reliability diagram).

Interview Question
Derive the gradient of the logistic regression loss. Why is it so clean?

Start with BCE: L = −[y·log(σ(z)) + (1−y)·log(1−σ(z))]. Using the fact that σ'(z) = σ(z)·(1−σ(z)), the chain rule gives: ∂L/∂z = −[y·(1−σ(z)) − (1−y)·σ(z)] = σ(z) − y = ŷ − y. It's clean because the sigmoid and the log are "natural partners" — the sigmoid is the canonical link function for the Bernoulli distribution, and cross-entropy is the natural loss for that distribution. Any exponential family distribution paired with its canonical link and natural loss produces a gradient of this form: (prediction − label).

Interview Question
What's the relationship between logistic regression and naive Bayes?

They're a generative-discriminative pair. Naive Bayes is generative — it models P(x|y) for each class and uses Bayes' theorem. Logistic regression is discriminative — it models P(y|x) directly. If the naive Bayes assumptions hold (features are conditionally independent given the class), both converge to the same decision boundary. But logistic regression is more robust when assumptions are violated because it directly optimizes the decision boundary without modeling the full data distribution. Naive Bayes needs less data to converge; logistic regression achieves lower asymptotic error.

Interview Question
Logistic regression vs SVM — when would you choose one over the other?

Both learn linear decision boundaries, but they optimize different things. Logistic regression minimizes cross-entropy (log-likelihood) and outputs calibrated probabilities. SVM maximizes the margin between classes and outputs distances, not probabilities. Choose logistic regression when you need probability estimates (ranking candidates, thresholding for different precision/recall tradeoffs), when interpretability of coefficients matters, or when you plan to use the model's confidence scores downstream. Choose SVM when you care only about the decision boundary (not probabilities), when you want to use the kernel trick for non-linear boundaries without explicit feature engineering, or in high-dimensional spaces where the margin-based objective can generalize better. In practice, logistic regression is more commonly used in industry because probabilities are almost always useful, and kernel SVMs don't scale well to large datasets.

Interview Question
Can logistic regression overfit?

Yes — and this trips up candidates who think of logistic regression as "too simple to overfit." It overfits in several scenarios: (1) High-dimensional sparse data — with many features relative to examples, the model has enough capacity to memorize noise. (2) Perfectly separable classes — if a hyperplane can perfectly separate the classes, the weights diverge to ±∞ trying to push probabilities to exactly 0 and 1. The loss keeps decreasing but the model becomes overconfident. (3) Many engineered features — polynomial feature expansion can blow up dimensionality. This is exactly why sklearn's LogisticRegression applies L2 regularization by default (C=1.0) — unregularized logistic regression is rarely what you want in practice.

3. The Bias-Variance Tradeoff

This is arguably the most important conceptual framework in ML. For any model, the expected prediction error decomposes as:

$$\text{Error} = \text{Bias}^2 + \text{Variance} + \text{Irreducible Noise}$$

Bias is the error from the model's assumptions — how far off the average prediction is from the truth. A linear model fit to a quadratic relationship has high bias. It will consistently miss the true pattern regardless of how much data you have.

Variance is how much predictions change across different training sets. A deep decision tree memorizes training data, so its predictions shift wildly with different samples. High variance = model is too sensitive to the specific training data.

Irreducible noise (Bayes error) is the inherent randomness in the data that no model can capture — measurement error, missing features, genuinely stochastic outcomes.

The Tradeoff in Practice

Simple models (linear regression, shallow trees) tend toward high bias, low variance. Complex models (deep trees, large neural nets, high-degree polynomials) tend toward low bias, high variance. The sweet spot is where total error is minimized — enough complexity to capture the true pattern, not so much that you fit noise.

Regularization, ensemble methods, and cross-validation are all tools for navigating this tradeoff. Regularization constrains complexity (reducing variance at the cost of some bias). Bagging (Random Forests) reduces variance by averaging. Boosting reduces bias by iteratively correcting errors.

Modern Nuance: Double Descent
The classical bias-variance tradeoff predicts a U-shaped test error curve as model complexity grows. But in modern deep learning, we observe "double descent" — test error decreases, increases (the classical peak), then decreases again as models become massively overparameterized. This happens because very large models find interpolating solutions that are smooth and generalize well. In an interview, mentioning this shows you're up to date, but frame it as: "the classical tradeoff holds for traditional models; in the overparameterized regime, there's interesting new understanding."
Interview Question
You have a model with high training accuracy but low test accuracy. Is this a bias or variance problem? How do you fix it?

This is a variance problem (overfitting). The model has learned training-set-specific patterns that don't generalize. Fixes: add regularization (L1/L2, dropout), reduce model complexity (fewer features, shallower tree, smaller network), get more training data, use data augmentation, apply early stopping, or use ensemble methods like bagging. The diagnostic: plot learning curves — if training error is low and validation error is high with a large gap, it's variance.

Interview Question
Both training and test accuracy are low. What's happening?

This is a bias problem (underfitting). The model is too simple to capture the underlying pattern. Fixes: use a more complex model, add more/better features, reduce regularization strength, train longer (if the model hasn't converged). Also check for data issues — label noise, data leakage in the wrong direction, or bugs in preprocessing can also cause universally poor performance.

Interview Question
How does ensemble learning relate to the bias-variance tradeoff?

Bagging (e.g., Random Forests) primarily reduces variance. By training many high-variance models on bootstrap samples and averaging, the variance drops roughly as 1/N while bias stays similar. That's why you bag deep trees (high variance) — averaging stabilizes them.

Boosting (e.g., XGBoost) primarily reduces bias. It starts with a simple model and iteratively fits residuals, building complexity gradually. Each round reduces bias. But it can increase variance if you boost too many rounds — that's why you need early stopping or regularization with boosting.

4. Regularization (L1 & L2)

Regularization adds a penalty term to the loss function that discourages large weights, preventing overfitting. The general form:

$$\mathcal{L}_{\text{total}} = \mathcal{L}_{\text{data}} + \lambda \cdot \Omega(w)$$

where λ (the regularization strength) controls the tradeoff between fitting the data and keeping weights small.

L2 Regularization (Ridge)

$$\Omega_{L2}(w) = \sum_j w_j^2 = \|w\|_2^2$$

L2 adds the sum of squared weights to the loss. Its gradient is 2λw — it's proportional to the weight itself. This means large weights get penalized proportionally more, pushing all weights toward zero but never exactly to zero. The result: weights shrink smoothly, multicollinearity is handled (the matrix XTX + λI is always invertible), and the model generalizes better.

L1 Regularization (Lasso)

$$\Omega_{L1}(w) = \sum_j |w_j| = \|w\|_1$$

L1 adds the sum of absolute weights. Its subgradient is λ·sign(w) — a constant push toward zero regardless of weight magnitude. This creates a qualitatively different effect: many weights are driven to exactly zero, producing a sparse model. L1 performs automatic feature selection.

Why Does L1 Produce Sparsity But L2 Doesn't?

The geometric intuition: the L1 constraint region is a diamond (with corners on the axes), while L2 is a sphere. The loss function's contours are more likely to intersect the diamond at a corner, where some coordinates are exactly zero. The sphere has no corners, so intersections almost always have all coordinates nonzero.

The gradient intuition: L2's gradient (2λw) shrinks toward zero but gets weaker as w approaches zero — it asymptotically approaches but never reaches it. L1's subgradient (λ·sign(w)) applies constant force regardless of magnitude, actually pushing weights through zero.

Elastic Net

Combines both: α·L1 + (1−α)·L2. You get L1's sparsity with L2's stability for correlated features. In practice, L1 arbitrarily picks one of a group of correlated features; Elastic Net tends to keep or drop them together.

The Bayesian Interpretation

Regularization has an elegant Bayesian justification. L2 regularization is equivalent to placing a Gaussian prior (mean=0, variance=1/λ) on each weight — you're saying "I believe weights are probably small." L1 is equivalent to a Laplace prior (mean=0, scale=1/λ) — the Laplace distribution has a sharp peak at zero, which is why it encourages exact sparsity. The regularized loss is the negative log of the posterior: L_data is the likelihood, Ω(w) is the prior, and MAP estimation gives you the regularized solution. This framing lets you justify regularization as "incorporating prior knowledge" rather than just "adding a penalty."

AdamW vs L2 — Know the Difference
Standard L2 regularization adds the weight penalty to the loss, so the penalty flows through the optimizer's moment estimates. AdamW decouples weight decay — it subtracts λw directly from weights after the Adam step, bypassing the adaptive learning rates. This matters because Adam's per-parameter scaling can suppress the regularization effect of L2. AdamW is the standard for training transformers. This is a common "show me you understand modern training" question.
Interview Question
When would you choose L1 over L2 regularization?

Use L1 when you believe many features are irrelevant and want automatic feature selection — the resulting sparse model is more interpretable and cheaper to serve. Use L2 when most features are useful and you want to keep them all but prevent any single feature from dominating. If features are correlated, L2 is more stable (L1 arbitrarily picks one from a correlated group). In practice: try Elastic Net if you're unsure, or use L1 first to identify important features, then retrain with L2 on the selected subset.

Interview Question
Do you regularize the bias term?

No. Regularizing the bias would penalize the model for shifting predictions up or down, which prevents it from fitting the mean of the target correctly. The bias captures the output's baseline level and has nothing to do with model complexity. All standard implementations exclude the bias from the penalty — and so should you.

Interview Question
How do you choose the regularization strength λ?

Cross-validation. Train models with different λ values (typically on a log scale: 0.001, 0.01, 0.1, 1, 10, ...), evaluate on a validation set, and pick the λ that minimizes validation error. Scikit-learn's RidgeCV and LassoCV do this efficiently using leave-one-out CV (for Ridge) or coordinate descent path (for Lasso). Never select λ based on training error.

Interview Question
What happens as λ → 0 and λ → ∞?

As λ → 0: the penalty vanishes, and you recover the unregularized model. For linear regression, this is OLS. You get minimum bias but maximum variance — the model fits training data as closely as possible, including noise. As λ → ∞: the penalty dominates, forcing all weights to zero (for both L1 and L2). The model ignores all features and predicts the mean of y (regression) or the majority class (classification) — it's the simplest possible "model." Maximum bias, zero variance. The optimal λ is somewhere in between, and you find it through cross-validation. This is the bias-variance tradeoff expressed through a single hyperparameter.

Interview Question
You have 500 features. How do you decide which ones to keep?

Three families of methods, each with different tradeoffs:

Filter methods — rank features independently of any model using statistical tests: correlation with the target, mutual information, chi-squared test (categorical), or variance threshold (drop near-constant features). Fast and model-agnostic, but they ignore feature interactions — two individually weak features might be powerful together.

Embedded methods — the model performs selection as part of training. L1/Lasso drives irrelevant weights to zero (as discussed above). Tree-based feature importance (Gini importance, permutation importance) ranks features by how much they contribute to splits. These capture interactions and are usually the best practical choice.

Wrapper methods — train models with different feature subsets and pick the best-performing one. Forward selection (add features greedily), backward elimination (remove features greedily), or recursive feature elimination (RFE). Most accurate but expensive — each evaluation requires training a model. Only feasible with hundreds of features, not thousands.

A practical pipeline: start with filter methods to remove obvious noise (zero variance, near-duplicate features), then use L1 or tree importance to narrow to a strong subset, and finally validate with cross-validation that removing features didn't hurt performance.

5. Gradient Descent

Gradient descent is the workhorse optimization algorithm for ML. The idea: iteratively update parameters in the direction that decreases the loss most steeply.

$$w \leftarrow w - \eta \cdot \nabla_w \mathcal{L}$$

where η is the learning rate and ∇L is the gradient of the loss with respect to weights.

Variants

Batch GD computes the gradient over the entire dataset. Stable but slow for large datasets — each step requires a full pass through the data. Memory-intensive: you need all data in memory simultaneously.

Stochastic GD (SGD) updates after every single example. Very noisy gradients, but each step is fast. The noise actually helps escape local minima and saddle points. In practice, the noise is often too much — convergence is erratic.

Mini-batch SGD is the practical middle ground. Compute the gradient on a batch of B examples (typically 32–512). You get parallelism from batch computation (GPU utilization), noise reduction from averaging, and frequent updates. This is what everyone uses in practice.

Learning Rate: The Most Important Hyperparameter

Too large: oscillate past the minimum or diverge entirely. Too small: take forever to converge and get stuck in sharp local minima. The optimal learning rate depends on the loss landscape, which changes during training — that's why learning rate schedules (warmup + cosine decay, step decay, etc.) and adaptive methods exist.

Learning Rate Schedules

Step decay drops the LR by a factor (e.g., ÷10) at fixed epoch milestones. Simple and still common for CNNs. Cosine decay smoothly anneals the LR following a cosine curve from peak to near-zero — the standard for transformer training, often paired with warmup. Reduce-on-plateau monitors validation loss and drops the LR when progress stalls — adaptive and requires no schedule tuning, good for experimentation. 1-cycle policy ramps up then down in a single cycle — can achieve faster convergence in fewer epochs. The general principle: start with a high enough LR to make progress, then decay it so the model can settle into a good minimum.

Adaptive Optimizers

SGD with Momentum adds a velocity term that accumulates past gradients: v ← βv + ∇L, then w ← w − ηv. This smooths out noisy updates, accelerates through flat regions, and helps escape shallow local minima. β is typically 0.9.

RMSProp keeps a running average of squared gradients and divides by √(avg), giving each parameter its own effective learning rate. Parameters with large gradients get smaller updates — this handles sparse features and different scales well.

Adam combines momentum (first moment) with RMSProp (second moment), plus bias correction for the early steps. Adam is the default optimizer for most deep learning tasks. Its main weakness: it can converge to sharper minima than SGD+momentum, which sometimes generalizes worse.

AdamW fixes Adam's interaction with weight decay (as discussed in the regularization section). This is the standard for training transformers.

Interview Pro Tip
If asked "which optimizer would you use?", a strong answer is: "AdamW with warmup + cosine decay for transformers and most deep learning. SGD + momentum with step decay for CNNs/computer vision where it's been shown to find flatter minima. Adam for fast prototyping. Always tune the learning rate — it matters more than the optimizer choice."
Interview Question
Why does SGD sometimes generalize better than Adam?

SGD + momentum tends to converge to flatter minima (regions where the loss doesn't change much if you perturb the weights), which generalize better to unseen data. Adam's per-parameter adaptive rates can guide the model into sharp minima (narrow valleys in the loss landscape) that have low training loss but are sensitive to distribution shift. The noise in SGD acts as implicit regularization, effectively averaging over nearby points. This is why for some tasks (image classification with CNNs), SGD still outperforms Adam on test accuracy despite being slower to converge.

Interview Question
What problems can gradient descent encounter?

Several: (1) Vanishing gradients — in deep networks, gradients shrink exponentially through layers, stalling learning in early layers. Fixes: ReLU activations, residual connections, proper initialization. (2) Exploding gradients — the opposite; gradients grow exponentially. Fix: gradient clipping. (3) Saddle points — in high dimensions, saddle points (not local minima) are the main concern; SGD noise and momentum help escape them. (4) Ill-conditioning — when the loss landscape is very elongated in some directions, GD oscillates. Fix: adaptive learning rates (Adam) or preconditioning.

Interview Question
What happens if you initialize all weights to zero?

Every neuron in a layer computes the same output, receives the same gradient, and gets the same update — forever. This is the symmetry problem: identical neurons stay identical because nothing breaks the symmetry. The network effectively has one neuron per layer regardless of width. The fix is random initialization, but the scale matters. Too large → activations explode through layers and gradients blow up. Too small → activations shrink to zero and gradients vanish. Xavier/Glorot initialization (Var = 2/(fan_in + fan_out)) keeps variance stable across layers and works well with sigmoid/tanh. He initialization (Var = 2/fan_in) accounts for ReLU zeroing out half the outputs and is the standard for ReLU networks. Biases are typically initialized to zero — they don't have the symmetry problem since each neuron's bias is independent.

Interview Question
Explain the learning rate warmup. Why is it needed?

Warmup gradually increases the learning rate from a very small value to the target rate over the first few hundred/thousand steps. It's needed because at initialization, the model's activations and gradients are unreliable — the loss landscape is poorly conditioned. Large learning rates at this point cause wildly unstable updates that the model may never recover from. Warmup lets the optimizer's statistics (momentum, second moment estimates) stabilize before taking large steps. It also sidesteps the bias correction problem in Adam — by the time the learning rate is large, the moments have converged naturally.

Interview Question
How does batch size affect training?

Larger batches: more accurate gradient estimates (less noise), better GPU utilization (more parallelism), but tend to converge to sharper minima that generalize worse. They also require learning rate scaling — the "linear scaling rule" says: if you double the batch size, double the learning rate. Smaller batches: noisier gradients act as regularization, tend to find flatter minima, but are slower per-epoch and underutilize GPU parallelism. The sweet spot depends on your hardware and dataset — 32 to 512 is typical. Very large batches (4K+) need careful LR warmup and sometimes LARS/LAMB optimizers to train well.

Interview Question
Is gradient descent guaranteed to find the global minimum?

It depends on the loss landscape. For convex problems (linear regression with MSE, logistic regression with BCE), yes — there's a single global minimum and gradient descent will find it (given a small enough learning rate). For non-convex problems (neural networks), no — the loss landscape has many local minima and saddle points. However, in practice this is less of a problem than it sounds. In high-dimensional spaces, most local minima have loss values very close to the global minimum (they're "equally good"). The real obstacles are saddle points — points where the gradient is zero but it's not a minimum. SGD noise and momentum help escape saddle points. The practical concern isn't "will I find the global minimum?" but "will I find a minimum that generalizes well?"

6. Overfitting & Underfitting

Overfitting means the model has memorized the training data, including its noise, and fails to generalize. Underfitting means the model is too simple to capture the underlying pattern. These are the practical manifestations of the bias-variance tradeoff.

Diagnosing Overfitting

The primary diagnostic is the gap between training and validation performance. If training loss is very low but validation loss is much higher, you're overfitting. Learning curves (performance vs. training set size) are the best visual tool: a large gap that doesn't close as data grows indicates overfitting; a high floor for both curves indicates underfitting.

Causes of Overfitting

Model too complex relative to data (e.g., a deep network on 500 examples), training for too many epochs, noisy or mislabeled data (the model fits the noise), data leakage (validation data leaks into training through preprocessing or feature engineering), and too many features with too few observations (curse of dimensionality).

Remedies for Overfitting

More training data is almost always the best fix when possible. Regularization (L1, L2, dropout, weight decay). Reduce model complexity (fewer layers/features, shallower trees). Early stopping — monitor validation loss and stop training when it starts to increase. Data augmentation — create synthetic training examples. Ensembling — average multiple models to reduce variance. Feature selection — remove noisy or irrelevant features.

Remedies for Underfitting

Use a more expressive model. Add more or better features. Reduce regularization. Train longer. Check for data problems (wrong labels, bad preprocessing, missing values handled poorly).

The Train / Validation / Test Split

Three sets, three purposes. The training set is what the model learns from. The validation set is for tuning hyperparameters (learning rate, regularization strength, architecture choices) and for early stopping — you make many decisions based on validation performance. The test set is touched exactly once, at the very end, to get an unbiased estimate of real-world performance. If you tune on the test set (even indirectly), you're effectively training on it, and your performance estimate is optimistic. A common split is 80/10/10 for large datasets. For smaller data, cross-validation replaces the fixed validation set.

Data Leakage
The most dangerous cause of seemingly great performance is data leakage — when information from the test/validation set bleeds into training. Examples: fitting a scaler on the full dataset before splitting, using future data as features in time-series, including the target (or a proxy) as a feature. In interviews, if a model seems "too good," always consider leakage first.
Interview Question
Your model's validation accuracy plateaus at 85% despite high training accuracy. You add more data but the gap doesn't close. What's happening?

If more data doesn't help, the issue might be: (1) Label noise — if 15% of your labels are wrong, 85% may be the ceiling. Check label quality. (2) Feature insufficiency — the features you have simply can't distinguish beyond 85%. You need better features, not more data. (3) Distribution mismatch — your validation set might come from a different distribution than training. (4) Data leakage inflating training metrics — check your preprocessing pipeline. The key insight: more data fixes variance problems, not bias problems or data quality problems.

Interview Question
How does dropout work and why does it help with overfitting?

During training, dropout randomly zeroes out neurons with probability p (typically 0.1–0.5) at each forward pass. This forces the network to not rely on any single neuron — it must learn redundant representations. The effect is similar to training an ensemble of all possible subnetworks. At inference, all neurons are active, and outputs are scaled by (1−p) to compensate for the higher activation — or equivalently, training uses "inverted dropout" that scales by 1/(1−p) during training so inference needs no change. Dropout acts as a stochastic regularizer that prevents co-adaptation of neurons.

Interview Question
What's early stopping, and how do you implement it properly?

Monitor validation loss after each epoch. If it doesn't improve for N consecutive epochs (the "patience"), stop training and restore the weights from the epoch with the best validation loss. Key details: (1) use a separate validation set that you never train on, (2) save the best checkpoint, don't just stop — the last epoch might not be the best, (3) choose patience based on how noisy your validation curve is (5–20 epochs is typical), (4) early stopping is effectively a form of regularization — it limits the model's effective capacity by limiting training time.

Interview Question
How does the number of features relative to training examples affect overfitting?

When features (d) approach or exceed the number of examples (N), overfitting becomes almost inevitable — this is the curse of dimensionality. With d > N, there are infinitely many perfect-fit solutions (the system is underdetermined), and the model will find one that memorizes training data but fails on unseen data. A rough rule of thumb: for stable linear models, you want at least 10–20× more examples than features. For neural networks, the picture is more nuanced — overparameterized networks can generalize well due to implicit regularization from SGD, but they still need enough data to learn meaningful patterns. The practical fix when d is large relative to N: regularization, dimensionality reduction (PCA), feature selection, or collecting more data.

Interview Question
Your dataset has 30% missing values in a feature. What do you do?

First, understand why the data is missing — this determines your strategy. Missing completely at random (MCAR): safe to drop rows, but wasteful. Missing at random (MAR): missingness depends on other observed features — imputation works well. Missing not at random (MNAR): missingness is informative (e.g., high-income people skip the income question) — you need to model the missingness itself.

Practical approaches: (1) Simple imputation — fill with mean/median (numerical) or mode (categorical). Fast but ignores relationships between features. (2) Add a missingness indicator — create a binary feature "is_missing" alongside the imputed value. This lets the model learn that missingness itself is a signal. (3) Model-based imputation — use KNN or iterative imputation (sklearn's IterativeImputer) to predict missing values from other features. More accurate but more expensive. (4) Drop the feature — if 90%+ is missing and the feature isn't critical, it may not be worth the imputation noise. For tree-based models, some implementations (XGBoost, LightGBM) handle missing values natively by learning optimal split directions for missing data — often the best option when available.

Interview Question
What are some data augmentation techniques, and when do they help?

Data augmentation creates synthetic training examples by applying label-preserving transformations, effectively increasing dataset size without collecting new data. It directly combats overfitting by increasing variance in the training set. Techniques vary by domain: Images — random flips, rotations, crops, color jitter, cutout/random erasing, mixup (blend two images and their labels). These are standard and nearly always help. Text — back-translation (translate to another language and back), synonym replacement, random insertion/deletion. More fragile — you can easily change meaning. Tabular data — SMOTE (for class imbalance, not general augmentation), mixup, noise injection. Less established and less reliably helpful than image augmentation — often regularization is more effective for tabular data. The key principle: augmentations must preserve the label. Flipping a cat photo still shows a cat, but flipping a chest X-ray might change the diagnosis.

7. Cross-Validation

Cross-validation is the standard technique for getting a reliable estimate of model performance and for tuning hyperparameters when you don't have a large enough dedicated validation set.

K-Fold Cross-Validation

Split the data into K equal folds (typically K=5 or K=10). For each fold: train on K−1 folds, evaluate on the held-out fold. Average the K performance scores. Every example appears in a test fold exactly once, and in training folds K−1 times. This gives a much more reliable performance estimate than a single train/test split.

Variants

Stratified K-Fold preserves the class distribution in each fold — critical for imbalanced datasets. If your data has 5% positives, each fold will also have approximately 5% positives. Always use stratified K-fold for classification.

Leave-One-Out (LOO) is K-fold with K = N (each fold has one example). It's nearly unbiased but has high variance and is computationally expensive. Useful for very small datasets. For Ridge regression, LOO CV has a closed-form solution, making it efficient.

Repeated K-Fold runs K-fold multiple times with different random splits and averages all results. Reduces variance of the estimate at the cost of more computation.

Group K-Fold ensures all examples from the same group (e.g., the same patient, the same user) end up in the same fold. Critical when examples within a group are correlated — without this, you get data leakage.

Time Series Split respects temporal ordering: each fold uses all previous data as training and the next chunk as validation. You never train on future data. This is essential for any time-dependent prediction task.

Nested Cross-Validation

When you use CV to both select hyperparameters and estimate performance, you're optimistically biased — you've selected the hyperparameters that work best on the same data you're using to estimate performance. Nested CV fixes this: an outer CV loop estimates performance, and an inner CV loop (within each outer fold) selects hyperparameters. The outer loop gives an unbiased performance estimate of the entire model selection procedure.

The #1 CV Mistake
Preprocessing before splitting. If you fit a StandardScaler on the entire dataset and then do cross-validation, the test fold's mean and variance have leaked into the training folds. Every preprocessing step that "learns" from data (scaling, imputation, PCA, feature selection) must be fit only on training folds and applied to the test fold. Use sklearn.pipeline.Pipeline to guarantee this.
Interview Question
Should you use cross-validation or a single holdout set? When?

Use CV when data is limited and you need a reliable performance estimate — 5-fold CV uses data 5× more efficiently. Use a single holdout (or even a separate test set) when data is abundant — with millions of examples, a fixed 80/10/10 split gives reliable estimates without the computational cost of multiple training runs. Also use holdout for computationally expensive models (large neural nets) where CV is prohibitive. In production ML, a common pattern: use CV during development/experimentation, then train on all data and evaluate on a held-out test set for the final model.

Interview Question
You're building a model to predict next week's user engagement on Prime Video. What cross-validation strategy do you use?

Time-series split — never standard K-fold. User behavior has temporal patterns (seasonality, trends, content release schedules). If you randomly split, your training set will contain future data, creating leakage. Use expanding or sliding window: train on weeks 1–4, validate on week 5; train on weeks 1–5, validate on week 6; etc. Also consider group-aware splitting so that all of a user's data stays in the same split, preventing user-level leakage. For Prime Video specifically, account for content release events — a new show launch changes the distribution, and your CV should test how the model handles that.

Interview Question
What's the difference between K-fold CV and bootstrapping?

K-fold partitions data into disjoint folds — each example appears in the test set exactly once. Bootstrapping samples N examples with replacement for training, and tests on the ~36.8% of examples not selected (the "out-of-bag" samples). Bootstrap has more bias (some examples appear multiple times, some never), but can give confidence intervals on performance. K-fold is the standard for model evaluation and hyperparameter tuning; bootstrapping is used more for estimating confidence intervals on statistics and is the basis for bagging in Random Forests. For model selection, K-fold is generally preferred.

Interview Question
How do you choose K in K-fold cross-validation?

K controls a bias-variance tradeoff of the CV estimator itself. Higher K (e.g., K=N, LOO): each training set is nearly the full dataset, so bias is low — but the K models are very similar to each other, so variance of the estimate is high. Lower K (e.g., K=2): each training set is only half the data, so bias is higher (the estimate reflects a model trained on less data than you actually have). K=5 or K=10 is the standard compromise — empirical studies show these give the best tradeoff. Use K=5 for large datasets or expensive models; K=10 when you have more compute. For very small datasets (<100 examples), LOO or repeated K-fold is worth the cost.

8. Feature Scaling & Normalization

Feature scaling transforms features to comparable ranges. It sounds like a minor preprocessing detail, but it has a major impact on model training and is a frequent source of interview questions — especially around when it matters and when it doesn't.

Standardization (Z-score Normalization)

$$x' = \frac{x - \mu}{\sigma}$$

Centers features to mean=0 and scales to unit variance. This is the default choice for most ML models. It doesn't bound the range (outliers can still produce large values), but it works well when features are approximately Gaussian.

Min-Max Normalization

$$x' = \frac{x - x_{\min}}{x_{\max} - x_{\min}}$$

Scales features to [0, 1]. Useful when you need bounded values (e.g., inputs to neural networks with sigmoid outputs, or image pixel values). The downside: it's very sensitive to outliers — a single extreme value compresses the rest of the data into a tiny range.

When Scaling Matters

Distance-based models (KNN, SVM, K-means) are directly affected by feature magnitudes. A feature ranging 0–1000 will dominate distance calculations over a feature ranging 0–1, even if the smaller feature is more informative. Scaling is essential here.

Gradient-based optimization (linear/logistic regression with GD, neural networks) converges much faster with scaled features. Unscaled features create elongated elliptical loss contours — the gradient points away from the minimum, causing oscillation. Scaled features create roughly circular contours where the gradient points directly toward the minimum.

Regularization requires scaling. L1 and L2 penalties treat all weights equally — but if feature A has range [0, 1000] and feature B has range [0, 1], then w_A will naturally be ~1000× smaller than w_B to produce similar-sized contributions. The regularizer will penalize w_B much more heavily, unfairly suppressing it. Scaling ensures the penalty is applied fairly across features.

When Scaling Doesn't Matter

Tree-based models (decision trees, random forests, XGBoost) are invariant to monotonic transformations of features. They split on thresholds, so only the rank ordering of values matters, not their magnitudes. Scaling a feature from [0, 1000] to [0, 1] doesn't change where the optimal split is. This is one of the practical advantages of tree-based models.

The normal equation for linear regression solves directly without iterative optimization, so feature scales don't affect convergence or the final solution.

Fit on Train, Transform on Test
Always compute scaling parameters (mean, std, min, max) from the training set only, then apply those same parameters to the validation and test sets. If you fit the scaler on the full dataset before splitting, statistics from test data leak into your training preprocessing. This is one of the most common and subtle forms of data leakage. Use sklearn.pipeline.Pipeline to enforce this automatically.
Interview Question
Should you standardize or normalize your features? How do you decide?

Use standardization (z-score) as the default — it's robust, works for most models, and doesn't assume bounded data. Use min-max normalization when the model expects bounded inputs (neural nets with sigmoid/tanh, or when you specifically need [0,1] range). Use robust scaling (subtract median, divide by IQR) when you have heavy outliers that would distort the mean and standard deviation. For sparse data (e.g., bag-of-words), don't center (it destroys sparsity) — use MaxAbs scaling instead, which divides by the max absolute value without shifting.

Interview Question
Your model uses a mix of tree-based features and a linear model in a stacking ensemble. Do you scale?

You don't scale for the tree-based models in the first layer — they're scale-invariant. But if the meta-learner (second layer) is a linear or distance-based model, you should scale its inputs — which are the outputs of the first-layer models. The key principle: scaling decisions are about the consumer of the features, not the producer. Each model in the pipeline only needs scaling if that specific model is sensitive to feature magnitudes.

Interview Question
How does feature scaling interact with regularization?

Without scaling, regularization is effectively applied unevenly. A feature with range [0, 1M] needs a tiny weight (~1e-6) to contribute meaningfully, while a feature with range [0, 1] needs a weight ~1. The regularizer penalizes both weights equally, so it disproportionately shrinks the large-weight feature. After standardization, all features contribute through similar-magnitude weights, so the regularizer penalizes model complexity fairly. This is why sklearn's Lasso and Ridge documentation recommends standardizing features before fitting — it's not optional with regularization, it's effectively required for correct behavior.

Interview Question
Do you scale one-hot encoded categorical features?

Generally no — one-hot features are already binary (0 or 1), so they're on a natural scale. Standardizing them would break their interpretability and produce non-intuitive values. However, there's a subtlety with regularization: if your continuous features are standardized (mean 0, std 1) but one-hot features are 0/1, the regularizer may still treat them unevenly depending on class frequency. A rare category (0.01 mean) has different variance than a common one (0.5 mean). Some practitioners standardize one-hot features for this reason, but it's not common practice. For ordinal encoded features (e.g., education level: 1, 2, 3, 4, 5), scaling may be appropriate if the model is distance-based or gradient-based, since the range depends on the number of categories.
